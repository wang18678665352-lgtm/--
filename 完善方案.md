# 电子医疗管理系统 - 功能完善方案

> 基于现有系统功能缺口分析，提出可落地的完善方案。
> 约束条件：纯 C 语言、链表实现、无第三方库、终端界面。

---

## 一、优先级划分

| 优先级 | 分类 | 说明 |
|--------|------|------|
| P0 | 核心缺失 | 当前业务流程断档，不改没法用 |
| P1 | 体验优化 | 现有功能可用但效率低，高频场景 |
| P2 | 管理增强 | 管理端缺失功能，提升运维能力 |
| P3 | 扩展需求 | 有更好，没有不影响核心流程 |

---

## 二、P0 — 核心缺失

### 2.1 医生排班系统

**现状**：系统中没有任何排班概念。患者预约时选了一个医生 + 日期，但系统不检查医生当天是否出诊。医生登录后也看不到自己的排班。

**方案**：

```
数据结构 (链表):
  ScheduleNode {
    schedule_id  char[20]    // 排班编号 SC20260502001
    doctor_id    char[20]    // 医生ID
    work_date    char[12]    // 值班日期 2026-05-02
    time_slot    char[10]    // 时段: 上午 / 下午 / 全天
    max_appt     int         // 预约号上限
    max_onsite   int         // 现场号上限
    status       char[10]    // 正常 / 停诊
  }
```

- 管理员端新增"排班管理"菜单：按周/按月批量生成排班、停诊登记
- 患者预约时筛选：只显示所选日期有排班且状态为"正常"的医生
- 医生端首页显示本人当日排班状态
- 数据文件：`data/schedules.txt`

**涉及修改**：admin.c（新增排班子菜单）、patient.c（预约时筛选排班）、doctor.c（首页展示）

---

### 2.2 候诊队列实时查看（患者端）

**现状**：现场挂号后患者只知道自己的排队号，看不到当前叫到几号了、前面还有几个人。

**方案**：

- 患者端"挂号状态查询"中增加候诊队列信息：
  - 当前叫号：XX 号
  - 您的排位：前面还有 N 人
  - 预计等待时间（基于 busy_level 估算）
- 医生端接诊时显示"当前叫号：XX"
- 数据来源：现有 `OnsiteRegistrationQueue` 即可支持，无需新增数据结构

**涉及修改**：patient.c（查询时增加队列位置计算）、doctor.c（接诊时显示当前叫号）

---

### 2.3 管理员后台新增医生/患者

**现状**：医生管理只有查看/修改/删除，没有新增。新医生只能通过注册页面创建，和线下录入场景不匹配。

**方案**：

- 医生管理菜单增加"新增医生"选项（case 4）
- 输入：姓名、用户名、职称、所属科室
- 逻辑：同时创建 User 记录 + Doctor 记录
- 患者管理同理增加"新增患者"

**涉及修改**：admin.c（新增 case 分支）、data_storage.c（可能需要增加 create_user 函数）

---

## 三、P1 — 体验优化

### 3.1 搜索与筛选

**现状**：所有列表只能分页查看，没有搜索/过滤。数据量大了以后找一条记录非常困难。

**方案**：

现有 `ui_search_list()` 已经支持搜索，但调用不够。需要补全的场景：

| 位置 | 添加搜索 |
|------|---------|
| 管理员-药品列表 | 按名称/ID 搜索 |
| 管理员-患者列表 | 按姓名/电话/ID 搜索 |
| 管理员-医生列表 | 按姓名/科室搜索 |
| 医生-开药选择药品 | 按名称模糊搜索（现有 smart_drug_input 已部分实现） |

**涉及修改**：各 list 函数改为调用 `ui_search_list` 而非 `ui_paginate`，或两者结合。

---

### 3.2 批量操作

**现状**：新增药品、排班等只能一条一条操作。

**方案**：

- 排班批量生成：管理员选择科室、起止日期、时段，一次性生成多天排班
- 药品批量补货：选择多个药品，统一输入补货数量
- 实现方式：循环遍历链表添加，不引入新数据结构

**涉及修改**：admin.c 排班管理、药品管理功能扩展

---

## 四、P2 — 管理增强

### 4.1 操作日志

**现状**：谁在什么时候删了医生、改了药价、取消了预约，完全不可追溯。

**方案**：

```
数据结构:
  LogNode {
    log_id       char[20]     // 日志ID
    operator     char[50]     // 操作人用户名
    action       char[20]     // 操作类型: create/update/delete
    target       char[20]     // 操作对象: doctor/patient/drug/ward/...
    target_id    char[20]     // 对象ID
    detail       char[200]    // 操作详情
    create_time  char[30]     // 操作时间
  }
```

- 在关键写操作后追加日志（新增、修改、删除、补货、取消预约等）
- 管理员端新增"操作日志查看"菜单，可按操作人/对象/时间筛选
- 日志文件只追加不覆盖
- 数据文件：`data/logs.txt`

**涉及修改**：admin.c、data_storage.c（新增日志读写函数）、在 doctor.c/patient.c 关键操作处追加日志调用

---

### 4.2 数据备份与恢复

**现状**：数据全是文本文件，误删或损坏就全没了。

**方案**：

- 管理员端新增"数据管理"子菜单：
  - 备份：将 `data/` 目录打包复制到 `data/backup/YYYYMMDD_HHMMSS/`
  - 恢复：列出备份列表，选择后还原
- 启动时自动检查数据文件完整性（文件是否存在、格式是否可解析）

**涉及修改**：admin.c、main.c（启动检查）

---

### 4.3 药品分类管理

**现状**：药品只有平铺列表，没有分类。药品种类多了以后难以管理。

**方案**：

- 药品结构增加 `category` 字段（西药/中成药/中药饮片/耗材...）
- 列表显示支持按分类筛选
- 数据迁移：现有药品默认分类为"西药"

**涉及修改**：data_storage.h（Drug 结构体）、admin.c 药品管理

---

## 五、P3 — 扩展需求

### 5.1 处方一次开多种药

**现状**：开药一次只能选一种药、输入数量、保存。需要多种药得反复操作。

**方案**：

- 接诊完成后进入"开具处方"流程时，支持循环添加药品
- 输入 0 结束添加，一次性保存整个处方
- 单次生成一个 Prescription ID，但包含多项 drug_id + quantity
- 风险：当前 Prescription 结构每条记录对应一种药，需要确认是否改为一条记录多种药，或维持现状但简化操作流程

**涉及修改**：doctor.c 的处方流程、data_storage.h 可能需要调整 Prescription 结构

---

### 5.2 医生工作量统计增强

**现状**：analysis.c 有医生负载统计，但只显示预约量 + 病历数。

**方案**：

在现有统计中增加：
- 本月接诊人数
- 本月开处方数 / 处方总金额
- 平均每患者就诊时长（估算）
- 与科室平均水平的对比

**涉及修改**：analysis.c，不涉及数据结构变更

---

### 5.3 急诊绿色通道

**现状**：紧急标识只是改个字段，没有真正的流程加速。

**方案**：

- 急诊患者挂号时自动排到队列最前面
- 急诊患者开药/检查时优先级标记
- 急诊患者自动床位预留逻辑
- 医生端首页醒目展示急诊患者

**涉及修改**：patient.c（挂号优先级）、doctor.c（首页展示）

---

## 六、实施路线

| 阶段 | 内容 | 预估工作量 |
|------|------|-----------|
| **Phase 1** | 医生排班系统、管理员新增医生/患者 | 3-4 个函数，1 个新数据文件 |
| **Phase 2** | 候诊队列查看、搜索筛选补全 | 现有函数增强，无新数据结构 |
| **Phase 3** | 操作日志、批量排班/补货 | 新数据文件 + 管理功能扩展 |
| **Phase 4** | 数据备份、药品分类、多药处方 | 视需求复杂度定 |

---

## 七、不改 / 已知不足

以下问题因约束条件（纯 C 终端、链表文件存储）暂不处理：

1. **图形界面/GUI** — 终端交互的局限性，不改
2. **网络/远程访问** — 单机系统，不改
3. **并发写冲突** — 文本文件不支持多进程同时写，不改
4. **数据库（SQLite）** — 当前无此硬性约束，不改
5. **实时消息推送** — 终端无法做到真正的实时推送（当前用轮询/手动刷新）
6. **密码复杂度校验** — 可做但优先级低
